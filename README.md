# Mini NPU Simulator

## 실행 방법

```bash
python main.py
```

실행 시 모드 선택:

1. 사용자 입력 (3x3)
2. data.json 분석
3. 패턴 생성기(보너스)
4. 보너스: 1차원 최적화 비교(보너스)

## 구현 요약

- Python 3.8+ 기반 콘솔 애플리케이션
- 외부 라이브러리 없이 구현 (`json`, `re`, `time` 사용)
- MAC 연산은 중첩 반복문으로 직접 구현
- 라벨 정규화
  - `+` → `Cross`
  - `x` → `X`
- 데이터 비교 정책
  - `abs(score_a - score_b) < 1e-9`이면 동점
- 모드 1 판정
  - `A 점수`, `B 점수`, `판정: A/B/판정 불가`
- 모드 2 판정
  - `Cross 점수`, `X 점수`, `판정: Cross/X/UNDECIDED`
- 케이스 실패는 프로그램 종료 없이 누적 처리

## 입력/검증 규칙

- 사용자 입력 모드(3x3)
  - 모든 입력은 줄마다 공백 구분 3개 숫자
  - 형식 오류 시: `입력 형식 오류: 각 줄에 3개의 숫자를 공백으로 구분해 입력하세요.`

- data.json 모드
  - 최상위 키: `filters`, `patterns`
  - 필터: `filters.size_5`, `filters.size_13`, `filters.size_25`
  - 각 크기 블록은 `cross`, `x` 키를 가짐
  - 패턴 키 형식: `size_<N>_<id>`
  - 각 패턴 항목: `input` (2차원 배열), `expected` (`+` 또는 `x`)

## 성능 분석

- 모든 크기에서 MAC 연산 시간을 최소 10회 반복 측정 후 평균(ms) 출력
- 표준 항목: `크기`, `평균 시간(ms)`, `연산 횟수(N^2)`

## 결과 리포트

테스트 실행 결과 총 6개의 테스트 케이스를 모두 통과하여 **0건의 실패**가 발생했습니다. 아래는 시뮬레이터 구현의 핵심 설계 및 이론적 원리 분석입니다.

### 1. 코드 구조 및 설계
- **MAC 로직 흐름 (`mac` 함수):** 입력 패턴과 필터 커널이 2차원 배열일 때, 행과 열을 순회하는 이중 for문을 돌며 동일 위치의 값을 곱하고(`*`) `total` 변수에 누적(`+=`)하여 점수를 반환합니다.
- **키 매칭 로직 설계 (`parse_size_from_key`):** 정규표현식(`re.match(r"^size_(\d+)_(?:.+)$", key)`)을 사용하여 `size_{N}_{id}` 형태의 키 문자열에서 데이터 크기 `N`을 동적으로 추출합니다. 이 `N`을 통해 `data.json`의 `filters.size_{N}` 항목과 필터 크기가 매칭되도록 설계했습니다.
- **정규화 분리 이유 (`normalize_label`):** 라벨 정규화 로직을 별도 함수로 분리함으로써 향후 새로운 형태(예: 세모, 네모)나 새로운 기호가 입력으로 들어오더라도 핵심 판단 로직의 수정 없이 정규화 함수만 업데이트하면 되는 유지보수성과 재사용성을 확보했습니다.

### 2. 핵심 기술 원리 적용
- **MAC 점수와 유사도 원리:** 패턴과 필터를 겹쳐 MAC 연산을 수행하는 것은 두 행렬의 내적(Dot Product)을 구하는 것과 동일합니다. 필터 상에 특정 패턴이 1로 표시되어 있으므로, 입력 패턴과 겹치는 위치가 1일 때 곱셈 결과가 1이 되어 누적 점수가 상승합니다. 따라서 형태가 완벽히 일치할수록 더 높은 점수가 도출되어 형태 유사도를 수학적으로 판별할 수 있습니다.
- **부동소수점 오차 원인:** 컴퓨터는 메모리에 실수를 저장할 때 2진수 부동소수점 방식을 사용합니다. 10진수 실수를 2진수로 변환하는 과정에서 무한 소수가 발생하면 한정된 비트에 맞추기 위해 값을 자르거나 반올림하게 되며, 이로 인해 아주 미세한 연산 오차가 발생합니다. 본 시뮬레이터에서는 이를 방어하기 위해 `EPSILON(1e-9)` 이하의 오차를 동점으로 간주합니다.
- **모드별 동점 처리 차이 이유:** 모드 1(사용자 입력)은 단순 A/B 비교이므로 인간 친화적으로 "판정 불가"를 출력하는 것이 UX 측면에서 타당합니다. 반면 모드 2는 주어진 정답(`expected`)과 시스템의 결과를 엄격하게 비교해 채점하는 자동화 테스트이므로, "UNDECIDED"라는 상수적 상태를 반환하여 확실한 실패(FAIL) 케이스로 식별 및 분류하기 위함입니다.
- **시간 복잡도 분석:** N×N 크기의 배열 2개를 MAC 연산하기 위해서는 모든 원소를 한 번씩 곱하고 더해야 하므로 총 연산 횟수는 N²번이 됩니다. 성능 분석 결과에서도 알 수 있듯 크기가 3, 5, 13, 25로 늘어남에 따라 $O(N^2)$ 복잡도에 기반하여 연산 시간이 기하급수적으로(제곱에 비례하여) 증가합니다. (측정은 I/O를 제외하고 순수 연산 함수 호출 구간만 측정함)

### 3. 심층 인터뷰 답변
- **새로운 라벨 확장 순서:** 1) `data.json`의 필터 스키마에 새로운 라벨(예: `triangle`)의 커널 데이터 추가 2) `normalize_label` 함수에 새로운 문자열 매핑 로직 추가 3) `classify_mode2`의 판정 로직에 새로운 라벨에 대한 점수 비교 분기(if-else) 추가 순으로 진행합니다.
- **대규모 패턴 처리 최적화 방안:** 2차원 배열을 1차원 배열로 평탄화(Flatten)하여 메모리 연속 접근을 유도, CPU 캐시 히트율(Cache Hit Ratio)을 높이는 방식으로 최적화가 가능합니다(보너스 과제로 구현됨). 상용 환경에서는 이중 for문 대신 SIMD 벡터화 연산, 멀티스레딩, 또는 GPU를 활용하여 병렬 처리 효율을 극대화해야 합니다.
- **CPU와 NPU 병목 설명:** CPU는 폰노이만 아키텍처를 기반으로 명령어를 순차적으로 처리(직렬)하기 때문에, 수억 번의 단순 곱셈/덧셈(MAC 연산)을 수행할 경우 연산 유닛보다 메모리에서 데이터를 가져오는 과정에서 극심한 병목이 발생합니다. 반면 NPU는 수천 개의 단순 MAC 연산 유닛을 칩 내에 병렬로 배치하여, 한 클럭 사이클에 대규모 데이터를 동시에 처리하도록 설계되었기 때문에 AI 병렬 연산에 최적화되어 있습니다.
